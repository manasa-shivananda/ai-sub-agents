## Problem Statement

Mansi is actively job hunting in Australia, targeting startups and Big 4 banks. She has 3 portfolio projects demonstrating basic Claude API usage (AI Code Reviewer, Document Q&A, Job Application Assistant). She needs a 4th project that demonstrates senior-level AI engineering: multi-agent orchestration, task delegation, failure handling, and production-grade patterns. The project should also be genuinely useful for her own job search.

## What Makes This Cool

A **live Agent War Room dashboard** where you can watch multiple AI agents analyze a job posting in real-time. Unlike existing job application tools (AIApply, JobCopilot, Jobright) that hide their AI behind a button, this project makes the orchestration visible. You see agents dispatched in parallel, watch failures and retries happen, see a Devil's Advocate agent challenge the other agents' conclusions, and get a final match report with confidence scores.

The trace output alone, in a GitHub README, is a portfolio artifact that gets interviews.

## Constraints

- Must demonstrate multi-agent orchestration, not just sequential API calls
- Must show production patterns: typed contracts, error handling, observability, graceful degradation
- Must actually work against real job postings (file/paste input for v1; URL scraping as stretch goal)
- Must be buildable in 1-2 weekends
- Must appeal to both startup CTOs (scrappy, ships fast) and bank architects (clean architecture, typed contracts)
- Budget-conscious: model tiering to minimize API costs
- Python (most common in Australian AI roles)

## Premises

1. **Primary audience is hiring managers, not job seekers.** Portfolio piece first, tool second.
2. **Observability IS the feature.** Make agent coordination visible — traces, logs, handoffs, decision points.
3. **Quality over quantity.** Deeply analyze one job at a time, not mass-apply.
4. **Claude API is the right foundation.** First-class sub-agent support, Mansi already knows it.
5. **This needs to actually work.** Real API calls, real job descriptions, real output.

## Cross-Model Perspective

An independent Claude subagent cold-read the session and highlighted:

- **Coolest unexplored idea:** A "Devil's Advocate" agent whose only job is to challenge other agents' conclusions ("You said strong match, but the JD says 5 years Go and the resume shows 0. Explain yourself."). Creates visible agent-to-agent negotiation — the thing no tutorial project shows.
- **Key insight:** Mansi's mention of "handle failures" reveals a systems thinker, not a demo builder. Failure handling should be a first-class visible feature.
- **50% existing:** OpenAI Swarm patterns + Claude API's native tool_use get you halfway. The other 50% is domain-specific agents, the observability layer, orchestrator decision logic, and output packaging.
- **Weekend prototype:** 5 Python files Saturday morning, observability layer Saturday afternoon, parallel execution + failure handling Sunday morning, polish Sunday afternoon.

## Approaches Considered

Pure Python CLI tool. 5 agents, async orchestration, execution traces printed to terminal. SQLite for trace storage. README with Mermaid diagrams.
- Effort: S (weekend)

- Effort: M (1-2 weekends)

- Cons: More scope, UI work might distract from agent engineering

### Approach C: Agent Framework + Job App as Example (Creative/Lateral)
Build a lightweight multi-agent framework, then implement job application pipeline as example.
- Effort: L (2-3 weekends)

## Recommended Approach

**Approach B: CLI + Live Trace Dashboard.** The live dashboard is the highest-impact portfolio artifact. It transforms "I built agents" into "watch my agents think." The CLI core keeps the architecture clean and testable, while the dashboard adds the visual wow factor that makes hiring managers stop scrolling.

## Architecture

### Agent Roster (5 agents + orchestrator)

```
┌─────────────────────────────────────────────────────────┐
│                    ORCHESTRATOR                          │
│  State machine: ANALYZE → GAPS+RESEARCH → ADVOCATE →    │
│  WRITE → COMPILE                                        │
│  Handles: dispatch, parallel coordination, retry,       │
│  graceful degradation                                   │
└──────┬──────────┬──────────┬──────────┬────────────┬────┘
       │          │          │          │            │
  ┌────▼────┐ ┌──▼───┐ ┌───▼────┐ ┌──▼──────┐ ┌──▼─────┐
  │  Job    │ │ Gap  │ │Company │ │Devil's  │ │Letter  │
  │Analyzer │ │Analyz│ │Research│ │Advocate │ │Writer  │
  └─────────┘ └──────┘ └────────┘ └─────────┘ └────────┘
       │          │          │          │            │
       ▼          ▼          ▼          ▼            ▼
  Structured  Gap report  Culture &   Challenges   Tailored
  JD JSON     w/severity  values      & rebuttals  cover letter
```

**Orchestrator State Machine:**

| State | Entry Condition | Agents Dispatched | Exit Condition | Failure Path |
|-------|----------------|-------------------|----------------|--------------|
| ANALYZE | Pipeline start | `job_analyzer` | Agent returns `JobAnalysis` | Abort pipeline (no JD = nothing to do) |
| SHORT_CIRCUIT_CHECK | ANALYZE complete | (none — orchestrator logic) | No dealbreakers found | If dealbreakers found → COMPILE with early-exit report |
| PARALLEL_RESEARCH | No dealbreakers | `gap_analyzer` + `company_researcher` (parallel) | Both return (or company degrades) | `gap_analyzer` fail → abort. `company_researcher` fail → continue degraded. |
| ADVOCATE | PARALLEL_RESEARCH complete | `devils_advocate` | Agent returns `AdvocateChallenge` | Skip advocate → proceed with uncontested scores |
| WRITE | ADVOCATE complete | `letter_writer` | Agent returns cover letter text | Retry 3x, then include match report without letter |
| COMPILE | WRITE complete (or short-circuit) | (none — orchestrator assembles) | `ApplicationPackage` produced | Always succeeds (assembles whatever is available) |

**Execution flow:**
1. `job_analyzer` runs first (sequential — everything depends on it)
2. Orchestrator checks for dealbreakers (security clearance, visa requirements, etc.)
3. `gap_analyzer` + `company_researcher` run in parallel (independent)
4. `devils_advocate` runs after gaps + company (needs both)
5. `letter_writer` runs last (needs all upstream)
6. Orchestrator compiles final `ApplicationPackage` from all agent outputs

### Agent Specifications

| Agent | Input | Output | Model Tier |
|-------|-------|--------|------------|
| Job Analyzer | Raw JD text (file or stdin) | Structured JSON: skills, requirements, level, red flags | Haiku (fast, cheap) |
| Gap Analyzer | Structured JD + resume | Matched skills with evidence, gaps with severity, transferable skills | Sonnet (reasoning) |
| Company Researcher | Company name | Culture, values, recent news, Glassdoor sentiment, interview tips | Sonnet (default: Claude knowledge; `--web-search` flag enables Brave Search API) |
| Devil's Advocate | All upstream outputs | Challenges to match assessment, counterarguments, risk flags | Sonnet (adversarial reasoning) |
| Letter Writer | All upstream outputs + advocate challenges | Tailored cover letter addressing top matches and preempting gaps | Sonnet (writing quality) |
| Orchestrator | Job URL + resume path | Coordinates all agents, handles failures, produces final report | Sonnet (coordination) |

### Inter-Agent Contracts (Pydantic)

```python
# models.py — typed contracts between agents

# --- Sub-models ---
class Skill(BaseModel):
    name: str
    category: str                          # "language", "framework", "concept", "tool"
    years_mentioned: int | None            # years required per JD, if stated

class SkillMatch(BaseModel):
    skill: Skill
    evidence: str                          # quote or reference from resume
    confidence: float                      # 0.0-1.0

class SkillGap(BaseModel):
    skill: Skill
    severity: str                          # "critical" / "moderate" / "minor"
    transferable: str | None               # alternative skill that partially covers this gap

class Challenge(BaseModel):
    claim: str                             # the claim being challenged
    argument: str                          # why it's wrong or risky
    severity: str                          # "high" / "medium" / "low"

class Rebuttal(BaseModel):
    objection: str                         # likely interviewer objection
    response: str                          # preemptive answer

class MatchReport(BaseModel):
    overall_score: float                   # 0.0-1.0
    matched_skills: list[SkillMatch]
    gaps: list[SkillGap]
    challenges: list[Challenge]
    recommendation: Recommendation

class TraceEntry(BaseModel):
    timestamp_s: float
    source: str
    target: str
    tokens_in: int | None
    tokens_out: int | None
    status: str                            # "dispatched" / "success" / "failed" / "retry"
    cost_usd: float | None
    error: str | None

class ExecutionTrace(BaseModel):
    entries: list[TraceEntry]
    total_duration_s: float
    total_tokens: int
    total_cost_usd: float
    agents_succeeded: int
    agents_failed: int
    retries_used: int

# --- Shared types ---
Recommendation = Literal["apply", "skip", "apply-with-strategy"]

# --- Agent result wrapper (handles success, failure, degraded mode) ---
T = TypeVar("T")

class AgentResult(BaseModel, Generic[T]):
    status: str                            # "success" / "failed" / "degraded"
    result: T | None
    error: str | None
    retries_used: int
    duration_s: float

# --- Top-level contracts ---
class JobAnalysis(BaseModel):
    title: str
    company: str
    required_skills: list[Skill]
    nice_to_have_skills: list[Skill]
    years_experience: int | None
    role_level: str                        # "junior" / "mid" / "senior" / "lead"
    red_flags: list[str]
    dealbreakers: list[str]                # triggers short-circuit (e.g., security clearance)
    raw_text: str

class GapReport(BaseModel):
    matched_skills: list[SkillMatch]
    gaps: list[SkillGap]
    overall_match_score: float             # 0.0-1.0
    recommendation: Recommendation

class CompanyProfile(BaseModel):
    name: str
    culture_summary: str
    values: list[str]
    recent_news: list[str]
    glassdoor_sentiment: str | None
    interview_tips: list[str]

class AdvocateChallenge(BaseModel):
    challenges: list[Challenge]
    rebuttals: list[Rebuttal]
    adjusted_match_score: float
    go_no_go: Recommendation

class ApplicationPackage(BaseModel):
    cover_letter: str
    match_report: MatchReport
    application_strategy: str
    trace: ExecutionTrace
```

### Observability Layer

Every agent call is wrapped by a tracer that records:
- Agent name, input/output token counts, latency
- Success/failure/retry status
- Cost estimate (per-model pricing)
- Inter-agent message hashes (for debugging)

Traces stored in SQLite and streamed via WebSocket to the dashboard.

**CLI trace output:**
```
[00.0s] ORCHESTRATOR → job_analyzer (1,247 tokens in)
[02.1s] job_analyzer → ORCHESTRATOR (832 tokens out, SUCCESS)
[02.1s] ORCHESTRATOR → gap_analyzer (2,079 tokens in)
[02.1s] ORCHESTRATOR → company_researcher (412 tokens in) [PARALLEL]
[03.8s] company_researcher → ORCHESTRATOR (FAILED, retrying 1/3)
[04.9s] company_researcher → ORCHESTRATOR (623 tokens out, SUCCESS)
[05.2s] gap_analyzer → ORCHESTRATOR (1,104 tokens out, SUCCESS)

```

### Failure Handling

- **Retry with backoff:** Each agent gets 3 attempts with exponential backoff
- **Graceful degradation:** If company_researcher fails all 3 retries, orchestrator continues without it and notes "company research unavailable" in the final report
- **Short-circuit:** If job_analyzer detects a dealbreaker (e.g., security clearance required), orchestrator stops early with explanation
- **Timeout:** Individual agent timeout of 30s, overall pipeline timeout of 120s

## Resolved Decisions

1. **Resume format:** Markdown. Easiest to parse, version-control, and include as sample data.
2. **Company research source:** Default to Claude's training knowledge (zero dependencies). `--web-search` flag optionally enables Brave Search API for fresher data. Interface is the same either way — agent returns `CompanyProfile` regardless of source.
3. **Job description input:** v1 accepts file path or stdin (paste). URL scraping (Seek.com.au via `httpx` + `beautifulsoup4`) is a Weekend 2 stretch goal. This keeps the core pipeline on track.
4. **Model costs:** ~$0.03-0.05 per job analysis. Acceptable for portfolio use.

## Open Questions

1. **Public repo considerations:** Resume data should be redacted or use sample data in the public repo. Need to decide on a realistic but non-personal sample resume.
2. **Rate limiting:** If running multiple analyses in sequence, should the orchestrator enforce a cooldown? Probably not needed for portfolio use.
